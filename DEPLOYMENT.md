# HELLO Deployment

This runbook assumes WordPress 7.0+, PHP 8.0+, a public HTTPS URL for your HELLO Bridge, and a Matrix bot account on the homeserver you want to use.

## 1. Run Your HELLO Bridge

The bridge is the only long-running Node process. Customer WordPress sites do not run it.

```sh
cd bot
npm install
cp .env.example .env
```

Set these values in `.env`:

```sh
MATRIX_HOMESERVER_URL=https://matrix.org
MATRIX_ACCESS_TOKEN=...
MATRIX_USER_ID=@hello-bot:matrix.org
BRIDGE_API_TOKEN=use-a-long-random-secret
BRIDGE_PORT=8787
IDENTITY_STORE_PATH=.data/identities.json
MATRIX_SYNC_STORE_PATH=.data/matrix-sync.json
SITE_REGISTRY_PATH=.data/sites.json
```

Start the bridge:

```sh
npm start
```

Put it behind HTTPS at a stable URL such as `https://hello.example.com`.

## 2. Install The WordPress Plugin

Build the plugin zip:

```sh
npm run package:plugin
```

Install `dist/hello.zip` in WordPress, or copy `hello/` into `wp-content/plugins/`.

Activate **HELLO**, then open **Settings > HELLO**.

## 3. Configure HELLO In WordPress

For plugin-only customer sites, set:

- `Connection mode`: Hosted bridge
- `Bridge URL`: your public bridge URL, such as `https://hello.example.com`
- `Bridge token`: the same value as `BRIDGE_API_TOKEN`
- `Webhook shared secret`: a long random secret generated by the plugin or pasted by the admin
- `Room alias prefix`
- sync direction
- moderation redaction behavior

The plugin sends its callback URL and webhook secret to the bridge when it creates a room. The bridge stores that mapping and uses it when Matrix messages need to become WordPress comments.

## 4. Configure Matrix For The Bridge

Create or choose a Matrix account for the bot. The bot account needs to be able to:

- create public rooms
- send messages
- join invited rooms
- redact events if you enable moderation redaction

The primary REST namespace is `/wp-json/hello/v1`.

## 5. Production Service

Run the bridge under a process manager such as systemd, launchd, PM2, or Docker. Preserve `.data/` across restarts.

### systemd Example

```ini
[Unit]
Description=HELLO Bridge
After=network.target

[Service]
Type=simple
WorkingDirectory=/srv/hello/bot
EnvironmentFile=/srv/hello/bot/.env
ExecStart=/usr/bin/node src/bridge.js
Restart=always
RestartSec=5
User=www-data

[Install]
WantedBy=multi-user.target
```

## 6. Smoke Test

1. Publish a new WordPress post.
2. Confirm the post edit screen shows a Matrix room in the **HELLO** metabox.
3. Open the public post and click **Join the discussion in Beeper**.
4. Send a Matrix message in the room.
5. Confirm the message appears as a WordPress comment.
6. Submit and approve a WordPress comment.
7. Confirm the WordPress comment appears in the Matrix room.
8. Mark one synced comment as spam or trash.
9. Confirm the Matrix event is redacted if redaction is enabled.

## 7. Operational Notes

- The WordPress plugin never stores plaintext reader email addresses.
- The bridge hashes onboarding email replies for Gravatar and stores the MD5 hash plus public profile metadata.
- First bridge startup stores the Matrix sync cursor and skips historical room messages to avoid importing old history by accident.
- The bridge stores room-to-site callback mappings in `SITE_REGISTRY_PATH`.
- The bridge rejects comment and redaction requests for rooms that were not created and registered through it.
- If the plugin cannot create or post to a Matrix room, the latest error is shown in the post metabox.
- If a Matrix user is kicked or banned by a moderator in a mapped post room, future messages from that user are sent to WordPress as held comments.

## 8. Direct Single-Site Mode

For development, the original single-site mode still exists:

```sh
npm run start:direct
```

That mode requires `WORDPRESS_BASE_URL`, `WORDPRESS_BOT_SECRET`, and `ROOM_REFRESH_MS`.
